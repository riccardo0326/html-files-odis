5.2.1. Group Data

The group data entry in the master data navigator (2) is managed by the group administrator.

DataDescription
LanguagesIn the languages editor, the target languages in the Volkswagen Group are created for the editing (see Section 7, “ODIS Creator Editing Process”) and order management (see Section 6, “ODIS Creator Order Management Process”) processes. In addition to the various language codes, you can specify if it deals with an editing language. Please note that for the ODIS Creator system to function correctly, the various language codes (DES code, PUB code, ISO code, LIVAS code) for communication must be configured correctly to other systems.
MarketsTarget markets, in which the Volkswagen Group is active, are created in the Markets editor. The predominant languages are assigned to the markets. A target market represents a market in which vehicles are sold. The language allocations for this represent the languages spoken in the market. By assigning a market to an object in the equipment network, the necessary translations are assigned for all objects connected to this object.
BrandsIn the Brands Editor, the individual brands with headquarters and worldwide locations within the Volkswagen Group are created.
RolesUser roles are created in the Roles editor. The label and description is can be freely edited, while the corresponding meta role is to be selected from the specified list.
Editing locationsEditing locations are created in the editing locations editor, the corresponding default markets are assigned, the editing tasks for the brand are defined and the respective teams are assigned. An editing locations represents a location at which the editorial contents of the selected brands are produced.
URL addressesInternet addresses, that reference documents online are managed in the URL address editor and editorial objects can be assigned.
Test languageThe test language modules are imported in the test language editor. The import will be canceled if the integrity of a test language module cannot be ensured. An error message will inform the user of this. The description, version number, creation date, creation time and the date and time of the import are given in addition to the module ID.
ViWi servicesIn the ViWi services Editor, ViWi services are listed and new services can be defined. It is only possible to define new services. Services cannot be deleted. When creating a new service, the name of the service must be defined and he JSON schema must be selected in the form of a JSON file. The schema files must have a name that is different from the files that have already been selected. The defined services can be “locked” or “approved” in the context menu. Only services that are “IN_PROGRESS” can be modified later. When saving the ViWi editor, the entries in the table will be validated (unique name, schema check of the ViWi file). If there is an error, it will not be possible to save.
JSON templatesIn the JSON templates editor, JSON templates are listed and new templates can be defined. It is only possible to define new templates. Templates cannot be deleted. When creating a new template, the name of the template must be defined and the JSON schema must be selected in the form of a JSON file. All entries in the list must have a unique name. The defined templates can be “locked” or “approved” in the context menu. Only templates that are “IN_PROGRESS” can be modified later. When saving the JSON template editor, the entries in the table will be validated (unique name, schema check of the JSON file). If there is an error, it will not be possible to save.
Remote repair target systemsThe target systems for Remote Repair can be specified in the Remote Repair target systems editor. They consist of a system name and display name for each. The system name must be unique in the system. The user will be informed if they try to give a duplicate name. In that case, it will not be possible to save. Target systems can be set to inactive using the checkbox so that they can no longer be selected. Target systems that are not used can be deleted. However, once they are used in a functional test, that will no longer be possible. If a target system is selected, the lower table called “Used by” will show in which function test the target system is used. The system name is retained when the data is exported later and cannot be changed if the table is edited again.
Remote repair requirementsThe requirements for Remote Repair can be specified in the Remote Repair requirements editor. If a new requirement is added, a positive whole number must be entered. Then the text for the requirement will be inserted using the text library. If the "Default” checkbox is activated, the value will automatically be set as a requirement in the function test in the Remote Repair area. Once the entries have been saved in the table, they can no longer be deleted. For existing entries, only the text of the requirement can be modified, but not the number.
Remote repair access parameter schema versionThe schema version for the Remote Repair access parameters can be specified in the Remote Repair schema version editor. The schema version to be used is entered as free text in the “Schema version” text field. The version information is included when the Remote Repair portion is exported. This is no longer relevant in the system and there is no schema check or validation when the editor is saved. The additional fields "Changed on" and "Changed by" are added by the system when saving.
