7.12.1. Importing XML templates

You can initiate the import of XML templates with the Import XML templates menu item in the administration view. Similar to the required authorization for the "Recompiling all function tests" functionality, both the roles of "author" and "location administrator" must be assigned to you simultaneously. When opening the menu item, a dialog to select XML files is first displayed to you. Here you can select one or more XML files. After confirming the dialog with OK, all selected files will be imported. During this, a corresponding editorial object is created for every XML template that is found. The file name without an ending is transferred as the system name. If the corresponding XML template has not existed in ODIS Creator until now, then the version will automatically be set to „1.0.0“. The file content is respectively saved as editorial object content and can be displayed through the editor context menu, the same as with the supplemental document.

If an XML template object with a matching system name already exists in ODIS Creator, then the system checks if the file to be imported has the same content as the file from the found XML template object. The content is compared symbol by symbol. If both files are the same, then this file will be skipped during the import. When the files are different, a new version is created from the found editorial object and the content is updated respectively. The main version number is increased by one, according to the schema (x+1).0.0. If the old version was „2.4.1“, then the new version will be „3.0.0“. After the import is finished, all XML templates that imported successfully will receive the Confirmed completed status.

You must check all objects for compatibility after the import and then risk-release them. This intermediate step makes it necessary that the import is done within the context of an order step, that you must set beforehand. The verification of XML templates in the assigned order step can also be performed by another author from the respective team; they do not necessarily need to have the role of „location administrator“. Before the XML templates are set to the Confirmed completed status, they will undergo a consistency check, like all editorial objects. This will check if the system name for editorial objects of the XML template type is unique across the system. Because no corresponding schema for the XML template structure exists, the content cannot be checked for validity. There will only be a check to see if a well-formed document is present. If the consistency check detects an error, then the editorial object will remain in the „in processing“ status and the import moves on to the next XML file. Because the author can create new XML templates, or revise existing ones, in ODIS Creator, the following scenarios may occur, which are unrelated to each other.

  • The XML template to be imported is in the system and only exists as a working version.

  • The XML template to be imported is in the system twice, meaning two separate XML template objects with the same system name are in ODIS Creator: one as a risk-released version and one as a working version without a previous version. This is possible, because a check for unique system names first occurs in the consistency check.

  • There are two separate XML template objects with the same name for the XML template to be imported in the system: one as a working version to a released version and one as a working version without a previous version. This scenario expands on a previous scenario.

To cover these scenarios, the following rules are applied to every file to be imported:

  • It will first look for a released version with the same system name. If this occurs, then a new version will be created for it and will be confirmed as completed.

  • If no release version is found, it will search for a working version that has an existing working version. It is irrelevant, if this working version is edited by a team that you are assigned to or not. If such a version is found, then this working version is deleted and a new version is created for the release version (solution for scenario 2 and 3). This is the only way to ensure that the version contains the correct content. If you currently have a working version locked for editing and because of this, you cannot delete it through the import, then the file is not processed and will be displayed as an error after the import.

  • Working versions found without any previous release versions are ignored, and a new editorial object is created and confirmed as completed by the import (solution to scenario 1).

If some files could not be imported, then an error message indicating which XML files could not be imported will be displayed after the import.
