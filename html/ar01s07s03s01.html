7.3.1. Object Search

By default, the object search runs within the action item that is currently selected in the Editing view. (See Figure 224, “Object Search”).

You can enter your specified search parameters in a table in the search dialog for the object search. The same as in the filter table, multiple search criteria can be combined in this table using the Boolean operators "AND" and "OR". You also have the option to combine the respective sections of the entered search criteria using parentheses. A search criterion always consists of a search attribute, an operator as well as a value. For this, the value can be set, based on the used attribute, by entering a text, entry selection in a selection field or through a dialog for date input. Placeholders can also be used for the search (see Section 7.3.1.4, “Search with Placeholders”). The available attributes may be different depending on the selected action element (see Section 7.3.1.8, “Search Parameter”). The table is always empty at first. To perform a search, at least one search criterion must be specified in the table. Only content from the currently set data language is searched through when there is language-dependent content. To limit the search, you have the option with the structure search (see Section 7.3.1.1, “Structure Search”) to search for objects within a specific object structure

Figure 224. Object Search


In addition to the searching within an action element, you have the option to search for multiple action elements at the same time. You must select the "Other action elements" checkbox for this. The search dialog layout will change from this selection. A group appears to select the action elements to be used for the search. Furthermore, the table for entering the search parameters of the initially selected action element is no longer displayed directly, but rather the tables of the various action elements are displayed in the tabs. You can select/deselect in which action elements should be searched (see Figure 225, “Object Search with Multiple Action Elements”). In the next step, you must define the search criteria for every selected action element. At least one search criterion must be entered for each selected action element. When searching for multiple action elements, you can also combine multiple criteria for each action element using the Boolean operators "AND" and "OR". The search occurs at the same time in the respective structure trees and all corresponding objects are displayed as a result (see Figure 231, “Search Results”).

7.3.1.1. Structure Search

For the structure search, the area of the searched editorial objects can be limited by a previous selection of one or more editorial objects. The object search can also include all of the used editorial objects in the selected editorial objects in all levels of the search.

When determining the selection, it must be determined if the "Mark" toggle button is active or not:

  • If the "Mark" toggle button is active, all editorial objects are applied to the search, which is selected in the navigation tree.

  • If the "Mark" toggle button is not active, all highlighted editorial objects in the navigation tree are applied to the search.

The system names of the selected editorial objects are displayed in the search dialog in a list box (see Figure 226, “Structure Search”).

Figure 226. Structure Search


The following options are given for the search in the selected editorial objects:

  • Same level: it will search exclusively in the selected editorial objects.

  • All levels, only primary objects: it will search in all primary objects of the current action element in all levels under the selected editorial objects.

  • All levels, primary and secondary objects: in addition to the primary objects, it will also search in the editorial objects of the other action elements in all levels under the selected editorial objects. It can also be selected with checkboxes here, which types of secondary objects should be searched. With the checkboxes on the first level, the types of secondary objects are set that are directly referenced by the primary objects. It can also be set on the second level, which editorial objects referenced by the function codes should be included in the search. These checkboxes are deactivated if the superordinate function code is deselected.

The option All levels, only primary objects is only available if all of the following conditions are met:

  • One or more editorial objects are selected in the current navigation tree in the editing view.

  • All selected editorial objects are primary objects in the current action element.

  • The option Other action elements is not selected.

If all the conditions are met, the option All levels, only primary objects is set by default when opening the search dialog. If the selected editorial objects are not all primary objects in the current action element, then the option All levels, primary and secondary objects is set by default.

With the search option All levels, primary and secondary objects, the types of secondary objects to be searched are set with option fields. For example, with the selection of one or more function codes with a restriction on the function code type, this can limit the search area to all accessed sub-programs on every level, and all measured value tables used by these function codes and preset measurements are excluded from the search.

7.3.1.2. Team/Brand Affiliation

Additional options fields can be used to set whether the search will be across brands or should be limited to the team or brand affiliation. It will such within the particular brand by default.

  • Cross-brand search: editorial objects from the responsible team at any location affiliated with another brand will also be included in the search results. The default filter for all action elements without a filter allocation is not used, so that no limitations relating to brand affiliation (through the default filter) are applied to it.

  • Limit to team affiliation: only editorial objects are included in the search results, which the current user's team is responsible for.

  • Limit to brand affiliation: using the team affiliation of the current user and their assigned location, it determines the amount of brands, in which the user is active. Only editorial objects are applied to the search results that are assigned to at least one of these brands. The brand relation is evaluated for this the same as for the deployment.

7.3.1.3. Search Results Filter

The filters currently set by the user in the user are applied during the object search. The filter currently set in the navigation tree will be applied respectively to each action element. If there are currently multiple navigation trees displayed for one action element, the last enabled filter will be used. For all action elements without a filter allocation, the default filter is used analogously to determining the subsequent nodes when opening the navigation tree.

7.3.1.4. Search with Placeholders

Placeholders can be used for the object search. The following placeholder symbols "*" and "#"are given. "*" serves as the placeholder for an arbitrary character sequence (also clear). For example, the search for the system name "engine*" will give objects with the systems names "engine control module", "engine electronics" and "engine_has_short_misfires" as the result.

The "#" placeholder stands for exactly one arbitrary character. The search for the vehicle project "AU34x_8#_2" will bring up the vehicle projects "AU34x_8L_2" and "AU34x_8N_2" as the result.

Due to performance reasons, placeholders should not be used at the beginning of the search, since this cannot use the index and the search could take longer.

7.3.1.5. Search in Translation-Relevant Texts

When searching in all translation-relevant texts, the entered character string will be searched in all language-dependent texts of the editorial objects. This affects the display names and the language-dependent XML content:

  • XML texts for function codes

  • Message texts for the measured values from measured value tables (no measured value names)

  • XML content from supplemental documents

The search is done in the data language currently set in the user interface.

The search is performed in the language-dependent XML content with the full-text search. It is not case sensitive. The search text can contain '*' as a placeholder.

Thereby, texts will not be found that are not directly included in the display names and the language-dependent contents, but rather are referenced by it:

  • Texts from the text library

  • Texts from default texts

  • Texts from master component lists

  • Texts from XML reference tables

  • Vehicle status texts and general system texts

The following texts may be found in FTE content:

  • Variable names and keyword names

  • Instructional comments

  • Development description

  • Names of accessed ASAM-Ecukom operations and the used parameters

  • LogicalLinks

  • Names of vehicle projects

  • Text marked as non translation-relevant (enclosed with ' symbol) consistently included in the function codes

  • Function expressions

  • Texts from error messages

  • XPATH expressions and their labels when reading XML templates

  • Areas and dimensions from progress indicators

  • VIN numbers in the set status

  • Non translation-relevant texts (enclosed with ' symbol) entered in dialogs

  • System names of diagnostic objects in the seat status suspicion rule list

Not found:

  • Used types of action elements

  • System names of used control module groups or versions

  • GUIDs of translation-relevant texts

  • IDs of used default texts, text IDs, vehicle status texts, general system texts ,etc.

  • Priorities and positions of error messages

  • Types of variables, keyboard entries

  • Indexes of select instructions

  • Lock status with the lock button

  • System name, document type, display type with document references

In summary, this means that texts are not found if they are between the '<' and '>' symbol in the FTE file, because these are not recorded in the full-text index on the index server. Texts that are between the '>' and '<' symbol are recorded by the index server in the full text index and will therefore be found.

Due to the different execution of the search in the display names and the search in the XML contents, small deviations in the search relationship are not to be avoided, if they yield different executions of the database search and the full-text search.

Examples: the input translation-relevant text includes a throttle valve control module will also find the text Adapt engine control module to throttle valve control module in display names. The text is not found using the full-text search in the message texts. To find the text "Adapt engine control module to throttle valve control module" using the full-text search in the message texts, the entry must be adjusted as follows: translation-relevant text contains *control* to *throttle valve*.

The entry translation-relevant text includes throttle* will also find the text Adapt engine control module to throttle valv. control module in display names. The text is not found using the full-text search in the message texts. To find the text "Adapt engine control module to throttle valve control module" using the full-text search in the message texts, the entry must be adjusted as follows: translation-relevant text contains throttle* *module.

7.3.1.6. Search in Non Translation-Relevant Texts

When searching in all non translation-relevant texts, the entered character string will be searched in all language-neutral texts of the editorial objects. This affects the metadata and the language-independent XML content:

  • FTE files for function codes

  • XML content from XML templates

  • XML content from preset measurements

The search in the metadata and the language-dependent XML content is performed with the full-text search. It is not case sensitive. The search text can contain '*' as a placeholder.

Examples: the entry non translation-relevant text contains Komp series finds XML templates with the Komp series content.

7.3.1.8. Search Parameter

When searching with the "LIKE" operator, entries with different capitalization will be found as opposed to searching with the operator "LIKE (case sensitive)".

Example: system name LIKE 01 engine electronics finds 01 engine electronics, 01 Engine Electronics, 01 ENGINE ELECTRONICS, etc.

If the "LIKE (case sensitive)" operator is used and contains no placeholder, then the search radius is replaced as the exact value using the high-performance operator "=". This operator considers capitalization. If searching for the system name "Drivetrain", the object will not be found with the search for "drivetrain".

Both operations con contain placeholder. When searching for parts of words, they must always be specified. A search for multiple words linked through OR, is no possible for these operations. Multiple lines must be linked with OR in the search dialog to do this.

Among other things, there are operators "contains" and "contains exactly". When using these operators, multiple words can be given as a search string. Explicit placeholders must be used for both operators, if parts of words should be searched for. For the operator "contains", the sequence is arbitrary and the separate words are linked with "or". For the operator "contains exactly", the words must be exactly in the specified sequence as the content.

The globally available search options (see Table 46, “Global Attributes”) are listed in the following, hereafter referred to as "attributes", that are offered regardless of the selected action element and can be selected. The attribute "Display name/title" is always called this. Depending on how it is used, the respective fields will be considered in the search.
