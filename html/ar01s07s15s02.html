ODIS creator supports use cases 1 through 4. In standard cases, use case 1 is used in which the open function test in ODIS Creator is run directly in the debugger without a vehicle connection and without diagnostic entry. Use cases 1 through 3 are described in the sections that follow. From the author's viewpoint, use case 4 for debugging traversal tests corresponds to debugging with default values.

7.15.2.1. Use Case 1: Debugging with Default Values

The "Debug with default values" use case is used for verification of a test program and all references within the editing environment with default value entry mode. The initiator is the author who has created or changed a function test. The function test must then be verified using default values.

NameContent
Use case:Verification of a test program and all references within the editing environment in default value entry mode.
Verification type:Internal
Actor:Author
Trigger:The author creates or changes the function test. The function test must then be verified using default values.
Scenario:The author is at his/her work station.
Requirement/context:There is a network connection to the VW network. ODIS Service does not need to be on the test computer. To start ODIS Debug, there is no requirement for diagnostic entry to run or communication with the control modules to be established.
Data used:ODIS Creator database.
Description/process:The test sequence is opened in the editing system and edited, if necessary. ODIS Debug starts and the author selects the "Start test module" debut mode in the ODIS Debug home screen. ODIS Debug plays back the test program sequence without restrictions (including all references). Default value entry must be possible at places in the test program where the debugger cannot access actual values. The author can edit the function test and all diagnostic objects and references again, and then save them and test again with the debugger.
Results:Verified test sequence including all references. The change to the function test is transferred back into the editing system database.

7.15.2.2. Use Case 2: Debugging with Actual Values

The "Debug with actual values" use case is used to verify a test program and all references using a real vehicle without diagnostic entry. The initiator is the author who has created or changed a function test. The function test must then be verified on the vehicle.

NameContent
Use case:Verification of a GFF program (including all references) in the editing system with a real vehicle without diagnostic entry and without ATP.
Verification type:Internal
Actor:Author
Trigger:The author creates or changes the function test. The function test must then be verified on the vehicle.
Scenario:The author is in the test workshop.
Requirement/context:There is a network connection to the VW network. There is a vehicle connection. Diagnostic entry was not performed. If diagnostic entry was performed (please note that there is no option to select a vehicle project in the ODIS Debug home screen because the "Vehicle project" option is deactivated), then you must first run the "Remove vehicle project" command in the "Extras" menu. Creating an ATP or performing diagnostic entry is not required to start ODIS Debug and debug using a vehicle. This debug type is selected and started directly in the ODIS Debug home screen.
Data used:ODIS Creator database.
Description/process:The test sequence is opened in the editing system and edited, if necessary. The author selects if the vehicle data from the production data or test baseline data should be used when debugging control module identification commands. The author starts ODIS Debug and select debugging with actual values in the ODIS Debug home screen (see Section 7.15.5, “Running the Debugger”). The author can select any vehicle project. Info: the projects available here for selection depend on the selection that was made regarding which data (production, test baseline or other sources) should be used for debugging. ODIS Debug plays back the test program sequence without restrictions (including all references). Default value entry is possible in places in the test program where the debugger cannot access actual data (such as information about the process, control module communication, measuring equipment instructions, system variables). The author can edit the function test and all diagnostic objects and references again, and then save them and test again with the debugger.
Results:Verified test sequence including all references. The change to the function test is transferred back into the editing system database.

7.15.2.3. Use Case 3: Debugging with Actual Values with Diagnostic Entry

The "Debugging with actual values with diagnostic entry" use case is used to verify a test program and all references with a vehicle connection. The initiator is the author who has created or changed a function test. The function test must then be verified on the vehicle.

NameContent
Use case:Verification in the editing system with a real vehicle including all references with diagnostic entry.
Verification type:Internal
Actor:Author
Trigger:The author creates or changes the function test. The function test must then be verified on the vehicle.
Scenario:The author is in the test workshop.
Requirement/context:There is a network connection to the VW network. The vehicle connection was established.
Data used:ODIS Creator database.
Description/process:The test sequence is opened in the editing system and edited, if necessary. The author performs diagnostic entry on the vehicle ("Start GFF" command in the "Extras" menu). The author can select in the vehicle features dialog which project should have diagnostic entry performed: automatic: the vehicle project is determined automatically based on the VIN and the vehicle features - manual: the author enters the VIN and the vehicle features and select the vehicle project. The author selects debugging with actual values in the ODIS Debug home screen. The ODIS Debug home screen provides the option of selecting a vehicle project, but because diagnostic entry was performed previously as described in this use case, there is no option for selecting another vehicle project on the screen. Note: to be able to select a different vehicle project in the ODIS Debug home screen, for example in order to debug based on a vehicle project as described in use case 2, a vehicle project must not be loaded. The author must remove the vehicle project in this location (by selecting the "Remove vehicle project" command in the "Extras" menu). The debugger plays back the test program sequence without restrictions (including all references). Default value entry is possible in places in the test program where the debugger cannot access actual data (such as information about the process, control module communication, measuring equipment instructions, system variables). The author can edit the function test and all diagnostic objects and references again, and then save them and test again with the debugger.
Results:Verified test sequence including all references. The change to the function test is transferred back into the editing system database.

7.15.2.4. Diagnostic Entry without ODIS Creator Database

When using vehicle models with a lot of vehicle data during diagnostic entry in ODIS Debug with ODIS Creator, larger amounts of data are loaded from the database. This process can take a long time, especially if there is a slow connection to the ODIS Creator database, such as in a workshop. For this reason, there is the option to run diagnostic entry in ODIS Debug without the ODIS Creator database. Diagnostic entry then takes place similarly to ODIS Service based on the production or test baseline that was specified in the debug configuration without loading newer information from the ODIS creator database.

There is still the option to perform diagnostic entry based on the ODIS Creator database. This allows you to continue working with the most current data as well as to debug the content from diagnostic entry.

In order for diagnostic entry to be able to run without the ODIS Creator database, make sure that you load the test baseline data regularly so that you are working with data that has been updated each day.

Diagnostic entry beings after the configuration dialog after executing the command "Start GFF", where you can specify if the ODIS Creator database should be used as the data source. It ends after the test plan is created. Current data is loaded again from the ODIS Creator database after diagnostic entry.
