##### <a id="Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Schreibe_XML"></a>7.14.17.3. Action Element - Write XML/XML IA

The "Write XML/XML IA" action element can be seen in [Figure 479, “Write XML/XML IA Action Element”](7-14-17-category-connection.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Schreibe_XML "Figure 479. Write XML/XML IA Action Element").

<a id="figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Schreibe_XML"></a>

![Image: Write XML/XML IA action element](images/fte_arbeitsflaeche_verbindung_schreibexml.png)

**Figure 479. Write XML/XML IA Action Element**

Here, you can export the vehicle data that was previously read into a work file that is defined by an XML template. Each required variable is assigned to an element in the XML template. You can also assign a text to an element. Then you transmit the work file with the "Send online" command.

XML templates can generally be divided into two groups: **IA templates** and **non-IA templates**. Non-IA templates may contain a maximum of one list element, and IA templates may contain one or more list elements (also nested inside one another). If an XML template is an IA template is determined solely by the presence of a processing instruction in the template content. The processing instruction is <?DMS\_BB IA\_TEMPLATE?>.

Both types of XML templates can be processed in this action element.

###### <a id="d4e25245"></a>7.14.17.3.3. Table

The table columns have the following meanings:

- Element: the elements of the XML template in which a text or a variable is entered. The names and number of elements displayed depends on the selection in the "XML template" field.
- Substitution (text or variable): here you can specify a value that will be transferred to the respective element.
- Interpretation/processing: you can switch between the options "Value / ---" and "File / tar + base64". The selection is optional.

  - Value / ---: the content of the ‘Substitution (text or variable)' column is interpreted as a value by the operating system. This is the standard case if nothing is selected in the ‘Interpretation/processing’ column.
  - File / tar + base64: the value of the string variables in the ‘Substitution (text or variable)’ column is interpreted as a file handle by the operating system and processed accordingly.

##### <a id="Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML-Dialog"></a>7.14.17.4. Action Element - Expand XML

The "Expand XML" action element can be seen in [Figure 480, “Expand XML Dialog Action Element”](7-14-17-category-connection.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML-Dialog "Figure 480. Expand XML Dialog Action Element").

<a id="figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML-Dialog"></a>

![Image: expand XML dialog action element](images/fte_arbeitsflaeche_verbindung_ergaenzexml.png)

**Figure 480. Expand XML Dialog Action Element**

Using this command, one or more related tags in a loop can be written multiple times in XML online packages. Without this command, unique tags would be needed for each control module, for example.

```
        *(<PARAM NAME="NAME_E1" VALUE="(Element_Ergaenzen_1)" />
        <PARAM NAME="NAME_E2" VALUE="(Element_Ergaenzen_2)" />
        <PARAM NAME="NAME_E3" VALUE="(Element_Ergaenzen_3)" />
        <PARAM NAME="NAME_E4" VALUE="(Element_Ergaenzen_4)" />
        <PARAM NAME="NAME_E5" VALUE="(Element_Ergaenzen_5)" />)*
        </DATA>
```

In contrast to the “Write XML/XML IA” command, in new files that are created based on XML templates, additional information is written into already existing XML files with the “Expand XML” command. As a prerequisite for correctly executing the “Expand XML” command, a "Write XML/XML IA” command must have already been performed in the process on the same working file and with the same XML template. Otherwise, runtime errors can result.

This command makes it possible to populate the sequence declared as a list in the XML template and add it into the working file. A new list item is written for each “Expand XML” command. It is not possible to make changes afterward to list elements that have already been added. All placeholders declared in the list sequence in the XML template must always be completely populated in the "Expand XML” command. Otherwise, it can result in invalid XML files in the process.

XML templates can generally be divided into two groups: **IA templates** and **non-IA templates**. Non-IA templates may contain a maximum of one list element, and IA templates may contain one or more list elements (also nested inside one another). If an XML template is an IA template is determined solely by the presence of a processing instruction in the template content. The processing instruction is <?DMS\_BB IA\_TEMPLATE?>.

The XML templates that can be used in the instruction must be from the “non-IA templates” group and may only ever contain exactly one defined list sequence.

##### <a id="Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML_IA-Dialog"></a>7.14.17.5. Action Element - Expand XML IA

The "Expand XML IA" action element can be seen in [Figure 481, “Expand XML IA Dialog Action Element”](7-14-17-category-connection.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML_IA-Dialog "Figure 481. Expand XML IA Dialog Action Element").

<a id="figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Verbindung.Ergaenze_XML_IA-Dialog"></a>

![Image: expand XML IA dialog action element](images/fte_arbeitsflaeche_verbindung_ergaenzexmlia.png)

**Figure 481. Expand XML IA Dialog Action Element**

Using this command, one or more related tags in a loop can be written multiple times in XML online packages. Without this command, unique tags would be needed for each control module, for example.

```
        *(<PARAM NAME="NAME_E1" VALUE="(Element_Ergaenzen_1)" />
        <PARAM NAME="NAME_E2" VALUE="(Element_Ergaenzen_2)" />
        <PARAM NAME="NAME_E3" VALUE="(Element_Ergaenzen_3)" />
        <PARAM NAME="NAME_E4" VALUE="(Element_Ergaenzen_4)" />
        <PARAM NAME="NAME_E5" VALUE="(Element_Ergaenzen_5)" />)*
        </DATA>
```

In contrast to the “Write XML/XML IA” command, in new files that are created based on XML templates, additional information is written into already existing XML files with the “Expand XML IA” command. As a prerequisite for correctly executing the “Expand XML IA” command, a "Write XML/XML IA” command must have already been performed in the process on the same working file and with the same XML template. Otherwise, runtime errors can result.

This command makes it possible to populate the sequences declared as lists in the XML template and add them into the working file. When doing this, note that the list sequences can be nested in any order. meaning lists inside of lists are possible. For this reason, both the values to be entered and the parameters of the XML paths must be specified when populating. The parameters of the paths then determine in which elements existing lists should be written. You can define them with concrete values or with variables. Entering the last parameter of a path is not necessary, because this is only there for the user’s information and is not relevant in the runtime execution.

When executing the “Expand XML IA” command, new elements are added to all lists/sublists in which at least one of the placeholders was populated with a value. The complete list sequences are always generated so that it is necessary to always completely populate all placeholders in a sequence with data in order to prevent runtime errors. Placeholders for defined lists do not always need to be populated with data. It is not possible to make changes afterward to list elements that have already been added. With subsequent “Expand XML IA” commands, only additional elements in lists/sublists can be added.

XML templates can generally be divided into two groups: **IA templates** and **non-IA templates**. Non-IA templates may contain a maximum of one list element, and IA templates may contain one or more list elements (also nested inside one another). If an XML template is an IA template is determined solely by the presence of a processing instruction in the template content. The processing instruction is <?DMS\_BB IA\_TEMPLATE?>.

The XML templates that can be used in the “Supplement XML IA” instruction are from the IA templates group and may contain any number of defined list sequences. List sequences can also be nested.

There is an add and delete function in the dialog so that additional lines can be added to/deleted from list placeholders in the table. These actions make it possible to add more than one list element in the same list within a single command. Lines that are added are only kept if they are populated with data. Deleting lines that are not populated with data does not have any other effect. To some extent, they will appear again when the dialog is opened again.

###### <a id="d4e25988"></a>7.14.17.19.8.3. Response parameter

Contains parameters in table form that must be evaluated. To define a response parameter, a JSON template must be selected from the drop-down list.

A response parameter consists of multiple parts:

- Name: the JSON name of the parameter
- Path: the path to the parameter within the JSON tree
- Type: the data type of the parameter
- Status: status of the parameter evaluation
- Variable: variable in which the value of the parameter should be written

Entering the path is mandatory and can be done either directly or selected using the “JSON response parameter selection dialog”.

The variable input is mandatory. The variable type must match the response parameter type.

The status of the parameter evaluation can have the following values:

- 0: parameter present and the data type of the associated received value is not compatible with the “Variable” data fields for the data set
- 1: parameter present and the data type of the associated received value is compatible with the “Variable” data fields for the data set
- -1: parameter is not present

If using the “JSON response parameter selection dialog”, the path will be calculated differently depending on the selection (see [Figure 521, “Path Calculation of the ViWi Response Parameter Selection Dialog”](7-14-19-category-flash.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Flash.Lese_ViWi_Request_Path "Figure 521. Path Calculation of the ViWi Response Parameter Selection Dialog")).

The view in the “JSON response parameter selection dialog” can be switched by pressing the “Expanded/reduced view” button. In this way, the element descriptions can be displayed or hidden. The "reduced view” is selected by default (see [Figure 522, “Reduced View of the ViWi Response Parameter Selection Dialog”](7-14-19-category-flash.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Flash.Lese_ViWi_Request_Param_Reduce "Figure 522. Reduced View of the ViWi Response Parameter Selection Dialog") and [Figure 523, “Expanded View of the ViWi Response Parameter Selection Dialog”](7-14-19-category-flash.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Flash.Lese_ViWi_Request_Param_Extend "Figure 523. Expanded View of the ViWi Response Parameter Selection Dialog")).

If response parameters have already been created, they can be reused using a convenience function when the JSON template is changed. When the service is changed, a request dialog appears that can be used to decide if the response parameters should be kept or cleared (see [Figure 524, “ViWi Read Response Convenience Function”](7-14-19-category-flash.md#figure.Funktionstesteditor.Arbeitsflaeche.Aktionselemente_Kategorie_Flash.Lese_ViWi_Request_Comfort "Figure 524. ViWi Read Response Convenience Function")).

The specification of the type is mandatory. If using the “JSON response parameter selection dialog”, the type will be calculated automatically and cannot be edited. If the path is entered manually, there will be a check to determine if the path is defined in the JSON schema of the selected JSON service. If the path is contained in the JSON schema, the type will be filled in automatically and it will not be possible to edit it. If the path is not contained in the JSON schema, the type will be deleted and must be selected manually using a combination box. The following values are offered for selection in the combination box.

- Integer
- Integer[]
- Number
- Number[]
- String
- String[]
- Boolean
- Boolean[]
