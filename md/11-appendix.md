## <a id="anhang"></a>11. Appendix

You will find legal information on open source software, the glossary for the explanation of terms, and an index attached.

### <a id="glossar"></a>Glossary

<a id="adhoc.auftrag"></a>**Ad hoc order**
:   Contains attributes entered by the author to complete an unscheduled order for editing one or more editorial objects.

<a id="administration.stammdaten"></a>**Administration master data**
:   Contains data, that is necessary for the organization of ODIS Creator usage, for the configuration of ODIS Creator or for location-specific adaptation of ODIS Creator.

<a id="baselines"></a>**Baseline**
:   Contains data that was prepared within the context of an export for distributing to downstream systems in a predefined structure.

<a id="didb"></a>**DIDB**
:   Contains information about the diagnostic database to use with ODIS Service.

<a id="diss-code"></a>**DISS Code**
:   Contains the structured coding of vehicle complaints.

<a id="documentum"></a>**Documentum**
:   Management software from the EMC Corporation to manage documents and content.

<a id="dms"></a>**Document Management System**
:   Document management systems provide the database-supported management of electronic document.

<a id="dtc"></a>**DTC**
:   DTC (Diagnostic Trouble Code) stands for the DTC memory objects in the ODIS project

<a id="eclipse"></a>**Eclipse**
:   Open source programming framework to develop software.

<a id="einheitliche.pruefsprache"></a>**Standard test language**
:   Under a standard test language, the test language description is understood in the ODIS project, which represents a program-technical description of the individual error search steps. In addition to the syntax and semantics of the programming language, the description of the function library and of the data model to describe the error search tree structure is also a part of the standard test language.

<a id="funktionstest"></a>**Function tests**
:   The diagnostic object function tests are imported that portray a specialization of the function test. They serve as the test of diagnostic objects.

<a id="hotfix"></a>**Hotfix**
:   Within the context of a hotfix action, it contains certain editorial objects for distributing to a specific dealer or a specific workshop.

<a id="HST"></a>**HST**
:   HST is an editing system to manage Technical Service Bulletins.

<a id="java"></a>**JAVA**
:   Object-oriented programming language that was developed by the company Sun Microsystems.

<a id="mutterliste"></a>**Master components list**
:   Table with component descriptions from wiring diagrams, component locations

<a id="offboard"></a>**Offboard**
:   Communication of the vehicle to the external tester (master)

<a id="onboard"></a>**Onboard**
:   Vehicle communication in "normal" operation, thus without an external trigger from a tester, for example.

<a id="redaktionsobjekt"></a>**Editorial object**
:   Editorial objects are editorial contents that can be created, changed and deleted. Editorial objects are grouped into order steps. On the other hand, order steps are components of an order.

<a id="redaktionssprache"></a>**Editing language**
:   Language in which the editors and authors create source-language documents.

<a id="redaktionssystem"></a>**Editing system**
:   Editing systems serve to manage content. A key feature of all editing systems is the separation of content, data structure and design. Editing systems are implemented in the industry for creating print media, catalogs and technical documentation.

<a id="rich.client"></a>**Rich client application**
:   Term from electronic data processing: for a client server architecture, the term fat client, rich client or also smart client is used for a client where the actual data processing is carried out by the client and also makes most of the graphic user interface available. The opposite of this is a thin client.
