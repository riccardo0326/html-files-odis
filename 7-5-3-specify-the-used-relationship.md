#### <a id="verwendet.beziehung.spezifizieren"></a>7.5.3. Specify the "Used" Relationship

At **knowledge base root notes**, the author can use the relationship to test programs to determine when these programs should be executed. Two time periods at the beginning of a session and three at the end are available for selection. The author can only select a maximum of one time period per group. The following can be selected:

- "Module start test",
- "Module start test (background)"
- “Module end test (before erasing DTC memory)”,
- ‘Module end test (after erasing/reading DTC memory and before calculating test plan)”,
- and “Module end test (diagnostic exit)”.

To do this, the author must select the corresponding attributes in the supplemental editor using the checkboxes. The module end test sequence above mirrors the sequence in the final workshop process when the module end tests are carried out. Variant rules can also be defined here. The **Start (background))** selection is accompanied with an enhanced consistency check. In start (background) modules, only one subset of all possible action elements may be used, because no user interaction is possible at this point in the process. The list of permitted elements is described in [Abschnitt 7.5.3.1, „Permitted Action Elements in Start (background) Modules“](7-5-3-specify-the-used-relationship.md#verwendet.beziehung.spezifizieren.Konsistenz "7.5.3.1. Permitted Action Elements in Start (background) Modules").

A **diagnostic object** can be linked to a control module description via drag and drop. The reference then appears in the used table. This can also be deleted again through the context menu in the knowledge base navigator.

The “Used” and “Used by” tables are loaded retrospectively when the editor is opened for the first time, when checking out and when displaying the history version. A placeholder text is displayed until the content of the tables is reloaded.

It is also possible to create new "used" relationships, so-called "external links", directly in the **diagnostic objects**. Using the ![](images/plus.png)- symbol at the "used" table will create a new, blank link. This is more accurately typified through the context menu from [Abbildung 264, „Create "Used" Relationship“](7-5-3-specify-the-used-relationship.md#figure.redaktion.wissensbasis.verwendetbeziehung.neu "Abbildung 264. Create \"Used\" Relationship"). These external links offer interfaces to other editing systems (LIVAS), databases (MOVE) or systems (EFA).

<a id="figure.redaktion.wissensbasis.verwendetbeziehung.neu"></a>

![Image: creating the "used" relationship](images/verwendet_beziehung_neu.png)

**Abbildung 264. Create "Used" Relationship**

  

Existing relationships can also be edited through the context menu. Variant rules can be defined at the diagnostic object to test program relationship. A separate variant rule can be specified for each of these relationships. This variant rule can be edited with the supplemental editor.

<a id="figure.redaktion.wissensbasis.verwendetbeziehung.bearbeiten"></a>

![Image: editing the "used" relationship](images/verwendet_beziehung_bearbeiten.png)

**Abbildung 265. Edit "Used" Relationship**

  

In the supplemental editor for diagnostic object to test program relationships, the attributes **sporadic error** and **without 12 V supply** can also be defined in addition to the variant rule.

The sporadic error attribute is now **NO** longer set in the test program test module! This flag can also be a mandatory field for certain brands (depending on the property set in Administration). If this is the case, the author must set a value for the flag for each connection between a diagnostic object and test program. In this case, flags that are not set prevent saving and prevent the completion message, except when there is no DTC reference to the associated diagnostic object. During completion, an error message will be entered in the consistency log. If the flat is not set for the affected brand, the input of a value for the **Sporadic Fault** attribute is optional. The diagnostic object can be saved with the value that is not set (**“-”**) and reported as completed. During preparation, the default value **sporadischer\_fehler="false”** (corresponds to the value “No”) can be selected in the exported XML documents in place of the value **“-”** that is not set.

The **"without 12V power supply"** attribute makes it possible to label the function tests so that the processing system can start the guided fault finding, even without the 12V power supply to the vehicle. If the 12V battery on the vehicle is removed during a repair (for example removal of the electrical motor in a high-voltage vehicle), having the flag not set prevents the tests from running further. No repair would be possible in this case. Whether the flag is set or not is shown in the “12V power supply” column using the value **not required** or **required**. For existing objects, the display shows **"-"** until the user has edited the usage editor for this relationship. During preparation, the default value **12V detection="false”** (corresponds to the value “required”) can be selected in the exported XML documents in place of the value **“-”** that is not set.

In the supplemental editor for relationships from diagnostic objects to supplementary documents, the author can also change the **reference category** initially adopted from the supplemental document used.

The **„access parameter“** attribute can only be changed in function tests that have the “Remote Repair” checkbox activated. To do this, the “...” button must be clicked in the corresponding cell of the function test. A dialog opens for setting access parameters (see [Abbildung 266, „Access Parameter Dialog“](7-5-3-specify-the-used-relationship.md#figure.redaktion.wissensbasis.aufrufparameter.remoterepair "Abbildung 266. Access Parameter Dialog")). If the function test contains access parameters, a new row can be added using “+” button or a row can be removed using “-”. When adding a new row, the name can be selected using a selection field. The type is transferred automatically and cannot be changed. Then a value must be assigned, because otherwise the dialog cannot be closed. Clicking the “OK” button transfers the values into the table cell and displays them one after the other, separated by commas. If the column width is too narrow, a tool tip will also be displayed when cursor is moved over the table cell. The display uses the schema “Name(type) = value”.

If an unpermitted action element is used, the knowledge base or function code cannot be confirmed as completed.
